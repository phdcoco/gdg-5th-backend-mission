## 코드의 유지보수

### 예외 처리

Check와 UnCheck 예외가 있다. 체크 예외는 컴파일 시점에 반드시 처리해야 하는 예외로 throws 또는 try-catch로 개발자가 반드시 처리해줘야 하는 예외이다. 언체크 예외는 런타임혹은 JVM에서 발생하는 예외로 주로 치명적인 오류에서 반환하는 예외이다. 자세한 건 백엔드 스터디 자료에…

비즈니스 예외 : 사용자의 잘못된 요청이나 도메인 규칙 위반 등, 복구와 예측이 가능한 예외이다. 예외 메시지도 API 응답에 그대로 전달할 수 있다. 직접 예외를 만들 수도 있다. 하지만 이펙티브 자바에서는 그러지 말라고 했다…

시스템 예외 : 애플리케이션 오류, 환경 문제 등, 복구 불가능하고 예측 불가한 예외이다. 클라이언트에게는 상세 메시지를 전달하면 안 되며 서버 내부 오류 수준만 보여줘야 한다.

@ControllerAdvice 를 통해 모든 예외를 한곳에서 처리할 수 있다. 컨트롤러 코드가 깔끔해질 수 있다. 예외를 표준화된 JSON으로 변환할 수 있다.

```json
{
  "code": "EMAIL_DUPLICATED",
  "message": "이미 가입된 이메일입니다.",
  "timestamp": "2025-11-30T12:00:00",
  "path": "/api/users"
}
```

### AOP

Aspect Oriented Programming, 관점 지향 프로그래밍이다. 어떤 로직을ㅇ 기준으로 핵심적인 관점, 부가적인 관점으로 나누어서 보고 그 관점을 기준으로 각각 모듈화한다.

Aspect : 흩어진 관심사를 모듈화한 것. 주로 부가기능을 모듈화한다.

Target : Aspect를 적용하는 곳. 메서드나 클래스 등이 있겠다.

Advice : 실질적으로 어떤 일을 해야할 지에 대한 것, 실질적인 부가기능을 담은 구현체이다.

JointPoint : Advice가 적용될 위치, 끼어들 수 있는 지점 메서드 진입 지점 등 다양한 시점에 적용이 가능하다.

PointCut : JointPoint의 상세한 스펙을 정의한 것.

스프링은 프록시 기반 AOP를 사용한다. 실제 객체 앞에 프록시 객체를 두고 메서드 호출 전, 후, 예외 시에 추가 기능을 실행할 수 있다. 

### 리팩토링 원칙

Clean Code 핵심 규칙 : 메서드의 이름은 짧고 의미 있게, 한 메서드는 하나의 역할, 이해하기 쉬운 변수명, 예측 가능한 흐름 등…

SOLID 원칙

- S : 단일 책임, 한 클래스는 하나의 책임만 자겨야 한다. 컨튼롤러에서는 비즈니스 로직 금지!
- O : 개방/폐쇄 원칙, 확장에는 열려 있고, 수정에는 닫혀 있어야 한다.
- L : 리스코프 치환 원칙, 자식은 부모 타입으로 정상 작동해야 한다.
- I : 인터페이스 분리, 큰 인터페이스 보다는 여러 개의 작은 인터페이스가 더 낫다.
- D : 의존 역전 원칙, 구현보다 추상(인터페이스 등)에 의존해야 한다.

DRY : Don’t Repeat Yourself, 중복된 로직, 코드, 상수를 제거하자.

KISS : Keep It Simple, Stupid, 불필요한 복잡성을 제거하자.

YAGNI : You Aren’t Gonna Need It, 미래에 필요할 것 같다고 미리미리 다 만들어두지 마라.
